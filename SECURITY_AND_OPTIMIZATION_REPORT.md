**Дата:** 2025-12-14

## 🔴 Критические уязвимости безопасности

### 1. API ключ в коде (AIAssistantView.swift:20)
**Проблема:**
```swift
private let apiKey = ""
```
**Риск:** Если ключ будет добавлен в код, он попадет в репозиторий и может быть скомпрометирован.

**Решение:**
- Использовать Keychain для хранения API ключа
- Или использовать переменные окружения через Info.plist (не коммитить в git)
- Добавить `.gitignore` для файлов с ключами

### 2. Отсутствие валидации URL (множественные места)
**Проблема:**
```swift
if let url = URL(string: "tel:\(phone)") {
    UIApplication.shared.open(url)
}
```
**Риск:** Если `phone` содержит специальные символы, может произойти инъекция URL.

**Решение:**
- Валидировать и экранировать параметры перед созданием URL
- Использовать `URLComponents` для безопасного построения URL

### 3. UserDefaults.synchronize() устарел (SettingsView.swift:570)
**Проблема:**
```swift
UserDefaults.standard.synchronize()
```
**Риск:** Метод устарел и не нужен в современных версиях iOS.

**Решение:** Удалить вызов `synchronize()` - данные сохраняются автоматически.

### 4. Отсутствие rate limiting для AI API
**Проблема:** Нет ограничения на частоту запросов к API.

**Риск:** Может привести к превышению лимитов API и дополнительным расходам.

**Решение:** Добавить rate limiting (например, максимум 10 запросов в минуту).

### 5. Нет валидации длины промпта
**Проблема:** Пользователь может отправить очень длинный промпт.

**Риск:** Превышение лимитов API, высокие расходы.

**Решение:** Ограничить длину промпта (например, 2000 символов).

## 🟡 Средние уязвимости

### 6. Отсутствие валидации email/phone перед использованием в URL
**Проблема:** Email и телефон используются в URL без дополнительной проверки.

**Решение:** Использовать `InputValidator` перед созданием URL.

### 7. Потенциальная утечка данных через логи
**Проблема:** Нет проверки на логирование чувствительных данных.

**Решение:** Убедиться, что логи не содержат PII (персональные данные).

## ⚡ Оптимизации производительности

### 1. @Query загружает все данные сразу
**Проблема:**
```swift
@Query(sort: \Appointment.date) private var appointments: [Appointment]
```
**Влияние:** При большом количестве записей загружаются все сразу в память.

**Решение:**
- Добавить пагинацию через `FetchDescriptor` с `fetchLimit` и `fetchOffset`
- Использовать `@Query` с предикатами для фильтрации на уровне БД

### 2. filteredClients вычисляется при каждом рендере
**Проблема:**
```swift
private var filteredClients: [Client] {
    // Фильтрация и сортировка в памяти
}
```
**Влияние:** При каждом изменении состояния пересчитывается весь список.

**Решение:**
- Кешировать результат
- Использовать `@Query` с предикатами для фильтрации на уровне БД
- Добавить debounce для поиска

### 3. Множественные фильтрации в памяти
**Проблема:** Фильтрация происходит в Swift, а не в базе данных.

**Влияние:** Медленная работа с большими списками.

**Решение:** Перенести фильтрацию в `@Query` с предикатами.

### 4. Отсутствие debounce для поиска
**Проблема:** Поиск выполняется при каждом изменении символа.

**Влияние:** Избыточные вычисления и возможные проблемы с производительностью.

**Решение:** Добавить debounce (например, 300ms).

### 5. StatsCache может быть оптимизирован
**Проблема:** `fetchLimit = 1000` и `fetchLimit = 5000` - слишком большие значения.

**Влияние:** Загрузка большого количества данных в память.

**Решение:**
- Использовать агрегатные функции БД вместо загрузки всех записей
- Оптимизировать запросы для расчета статистики

### 6. Большие списки без виртуализации
**Проблема:** `LazyVStack` помогает, но можно улучшить.

**Решение:** Убедиться, что используется правильная виртуализация.

## 🔧 Примеры исправлений

### Исправление 1: Валидация URL
```swift
func safePhoneURL(_ phone: String) -> URL? {
    guard InputValidator.isValidPhone(phone) else { return nil }
    let cleaned = phone.replacingOccurrences(of: "[^0-9+]", with: "", options: .regularExpression)
    return URL(string: "tel:\(cleaned)")
}
```

### Исправление 2: Debounce для поиска
```swift
@State private var debouncedSearchText = ""
.onChange(of: searchText) { _, newValue in
    Task {
        try? await Task.sleep(nanoseconds: 300_000_000)
        debouncedSearchText = newValue
    }
}
```

### Исправление 3: Rate Limiting
```swift
private var lastRequestTime: Date = .distantPast
private let minRequestInterval: TimeInterval = 6.0 // 10 запросов в минуту

func getResponse(...) async -> String {
    let now = Date()
    guard now.timeIntervalSince(lastRequestTime) >= minRequestInterval else {
        return "Please wait before making another request."
    }
    lastRequestTime = now
    // ... остальной код
}
```
